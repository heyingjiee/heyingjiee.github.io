面向对象设计 ​

众所周知，JS本身并不是严格的面向对象语言，class关键字本质上只是语法糖，而且并不健全。但是，随着TS越来越流行，我发现TS对于面向对象的支持很健全，是时候研究下面向对象编程了

但是，如何设计一个面向对象的复杂系统？这是一个一直困扰我的问题

在我记忆中，大学课本《数据库原理》中提到过的ER图，通过绘制ER图描述实体以及他们的关系，再根据一些特定规则就能转化为数据的表结构（符合第三范式、BC范式），即通过ER图就可以来设计数据库结构

是否存在某种图可以描述类结构与类之间的关系，然后通过某些特定规则转化为更优的结构或关系？

很遗憾，到目前为止我并没有找到类似的规则，至于原因，我推测关系型数据库是基于数学的集合论而来的，符合严谨的数学推理，可以通过某些规则做等价运算，而类是基于一种思想，其并不是基于数学推理而来

不过，虽然没有找打固定的规则做转化，但是对于面向对象的设计我还是找到了一些学习思路

  • UML类图：用于描述面向对象系统的架构（实体+关系）
  • 设计原则：设计实体之间关系的设计原则
  • 23种设计模式：实际场景中，不可能完全遵循所有设计原则。实体之间使用何种关系也有取舍。所以（符合某些原则的类/接口+某些合适的关系）组成了设计模式

UML类图 ​

UML，全称为Unified Model Language，即统一建模语言，是由一整套图表组成的，为面向对象系统的产品进行说明、可视化和编制文档的一种标准语言

UML具有许多不同类型的图表，包括：

  • 静态图：用例图、类图、包图
  • 动态图：活动图、状态图、时序图、协作图

这些不同的图，可以提供从不同的角度来描述整个系统，这篇文章只针对类图（其用来描述整个系统的结构），

学习目标：希望能通过设计类图，进行整个系统的架构

代码实现：使用TS，部分内容使用Go语言（后期也会补充TS实现。虽然Go使用struct来构造结构，严格来说并不是面向对象的，但是核心思想是一样的）

UML类图整体分为两大部分：

1、实体：类 、接口

2、关系：继承（也叫泛化）、实现、组合（合成）、聚合、依赖、关联

继承 ​

在代码中通常体现为A类派生自B类，子类B继承了父类的属性和方法，从而可以重用父类的代码，并在其基础上添加或修改功能

ts
class ParentClass {
  	name:string;
    constructor(name: string) {
      this.name=name
    }

    sayHello() {
        console.log(`Hello, ${this.name} from ParentClass`);
    }
}

class ChildClass extends ParentClass {
    constructor(name: string) {
        super(name); // 调用父类构造函数
    }

    // 子类可以继承父类的方法，也可以添加自己的方法
    sayHelloFromChild() {
        console.log(`Hello, ${this.name} from ChildClass`);
    }
}

// 创建子类的实例
const childInstance = new ChildClass("Alice");

// 可以访问从父类继承的方法
childInstance.sayHello(); // 输出：Hello, Alice from ParentClass

// 也可以访问子类自己的方法
childInstance.sayHelloFromChild(); // 输出：Hello, Alice from ChildClass

依赖 ​

依赖关系是一种弱化的关联关系，依赖是短暂的、临时性的，所以依赖关系的两个类，依赖方不持有被依赖方的实例

ts
class Utils {
	//xxx
}

class Monitor {
  	errorQueue=[];
    observeError(utils: Utils) {
        // 一般是使用Utils对象处理内部的数据，与`关联`的区别是：类内部并不存储Utils的实例
      	// 这里Utils的函数能向内部写入数据，这个例子中每次出现错误errorCallBack内部都触发一个回调，就会向errorQueue写入数据。这个思路很有意思，Monitor依赖于Utils的回调。
        // 这样把错误处理外置到了Utils模块，Monitor只负责存储，这样就把类的功能分离了
      
      	Utils.errorCallBack(this.errorQueue)
    }
}

const utils = new Utils();
const monitor = new Monitor();

monitor.observeError(utils);

总结 ​

继承、实现不必多说，是面向对象语言最常见的关系

组合：A对象每部持有B对象，其生命周期一致

聚合、关联：A对象每部持有B对象，但是A销毁，B可独立存在

依赖：A内部不持有B对象，仅仅是使用

UML的设计原则 ​

单一职责原则(S) ​

单一职责原则（Single Responsibility Principle，SRP）：一个类应该有且只有一个引起它变化的原因

里氏替换原则(L) ​

里式替换原则（Liskov Substitution Principle，LSP）：所有引用基类的地方必须能透明地使用其子类的对象

简单地说：所有父类能出现的地方，子类就可以出现，并且替换了也不会出现任何错误

这句话暗含了几层含义：

  • 子类必须完全实现父类的方法。如果子类不能完全实现父类的方法（例如：覆写某些方法），就要断开父子继承关系，替换为依赖、组合、聚集关系

    推荐独立出来

  • 里氏原则反过来不成立，子类出现的地方，替换为父类不行

    调用方法时，应使用继承链条中有对应方法的父类作为入参类型，子类会继承完全实现父类方法

    ts
    // 这样函数就可以传入子类了，子类也有getName方法
function setGunType(instance:parentType){
  parentType.getName() //
}

里氏替换由于子类只能在父类的基础上增加方法，而不能覆写，这样就增强程序的健壮性，在迭代逻辑的时也可以很好的保证兼容性

依赖倒置原则(D) ​

依赖倒置原则（Dependency Inversion Principle，DIP）：高层模块不应该依赖底层模块，两者都应该依赖其抽象

核心：面向接口编程，即各个类通过实现有依赖关系的抽象接口，达到解耦类之间依赖的目的。这种依赖可以是：实体的组合、聚合、关联、依赖关系

以组合关系为例，下面是司机、奔驰车两个类的组合关系，整体依赖部分，对于一个复杂系统大量的依赖关系会出现牵一发而动全身的严重问题

例如，让司机去驾驶宝马，但是由于driver入参类型为Benz（依赖Benz），所以不能只依赖Benz

但是，如果将类之间的依赖关系转变为：接口或抽象类之间的依赖 ，类去实现这些有依赖关系的接口或抽象类，从而实现一种间接的依赖关系（也称为接口注入）

最佳实践

  • 不超过两层的继承都是可以忍受的。一旦超过，就建议还是从接口或抽象类派生子类，这是依赖倒置的基本要求
  • 变量的表面类型尽量是接口或者是抽象类（符合里氏替换原则）
  • 抽象类与接口不同，抽象类的方法可以有具体的实现，注意派生的子类不能覆写已实现的方法（符合里氏替换原则）

接口隔离原则(I) ​

接口隔离原则（Interface Segregation Principle，ISP）:类间的依赖关系应该建立在最小的接口上。

核心：上面提到了面向接口编程，ISP原则表达的就是接口的内容要尽可能小。类可以根据其实际需要的功能，实现多个小接口，而不是去实现一个十分复杂的大接口

最佳实践

  • 保证接口的的内容要尽可能小。但是，设计粒度越小同时也带来了结构的复杂化，这个“度”如何来判断呢？需要根据经验和常识判断，没有一个固化或可测量的标准
  • 接口要高内聚。尽量少的public方法
  • 已经被污染了的接口，尽量去修改，若变更的风险较大，则采用适配器模式进行转化处理

迪米特原则 ​

迪米特原则（Law of Demeter，LoD）也称为最少知识原则（Least Knowledge Principle，LKP）：一个类应该对自己需要耦合或调用的类知道得最少

核心：每个对象都必然会与其他对象有耦合关系，但是迪米特原则要求类只和他的朋友类交流，且要保持距离（不能过分耦合）

朋友类：出现在成员变量、方法的输入输出参数中的类称为成员朋友类，而出现在方法体内部的类不属于朋友类

1、只和朋友交流

开放封闭原则(O) ​

开放封闭原则(Open-Closed Principle，OCP）：类对扩展开放，对修改关闭

最常见的实现是，通过接口或者继承，当增加或修改功能时不必变更主体的逻辑，而是通过接口实现扩展新的逻辑

下面是一个书店的例子：

现在新增一个需求：要求小说进行打折活动，售价打九折。修改思路：

  • 修改接口：即修改IBook接口的getPrice方法使其返回原价的九折

    不推荐这种，因为接口类似于一种契约，一旦修改接口实现该接口的类都会出现变动。除非确实是所以类的统一改动或者大规模重构，否则不能修改。

    这个例子了，一旦修改了getPrice为返回售价的九折，实际上就丢失了获取真实价格的方法

  • 修实现类：即修改NovelBook类的getPrice方法使其返回原价的九折

    不推荐这种，与上面原因类似，NovelBook会丢失了获取真实价格的方法

  • 推荐：扩展修改

    扩展子类OffNovelBook，当某方法获取价格时，入参类型为NovelBook，传入OffNovelBook实例。未来如果下线打折业务，则替换为NovelBook实例即可（里氏替换原则）

上面的例子是修改，如果是新增呢？相比于新增一个实体类，更推荐创建一个新的接口去继承IBook接口

工厂模式 ​

工厂方法模式的扩展性非常优秀。在增加产品类的情况下，只要适当地修改具体的工厂类、扩展一个工厂类，就可以完成

ts
// 定义产品接口
interface Product {
  operation(): void;
}

// 具体产品类A
class ConcreteProductA implements Product {
  operation(): void {
    console.log('ConcreteProductA operation');
  }
}

// 具体产品类B
class ConcreteProductB implements Product {
  operation(): void {
    console.log('ConcreteProductB operation');
  }
}

// 工厂类
class Factory {
  static createProduct(type: string): Product {
    switch (type) {
      case 'A':
        return new ConcreteProductA();
      case 'B':
        return new ConcreteProductB();
      default:
        throw new Error('Invalid product type.');
    }
  }
}

// 使用工厂创建产品
const factory = new Factory();
const productA = factory.createProduct('A');
const productB = factory.createProduct('B');

productA.operation(); // Output: ConcreteProductA operation
productB.operation(); // Output: ConcreteProductB operation

策略模式 ​

策略模式类图:实现IStrategy接口的三个策略。Context类构造函数入参类型是策略接口，赵云类直接在其函数中new Context(一条策略)，Context实例调用operate执行内部的策略实例的operate

注意：选择使用哪个策略是由高层模块决定的

优点：扩展性好，扩展策略只需要实现策略接口即可

缺点：每个策略都是一个类，会导致策略类数量过多难以维护（策略数量超过4个，则需要考虑使用混合模式=>策略模式+工厂模式。工厂模式选择使用哪个策略）

适配器模式 ​

公司突然需要引入外包员工，可是外包员工的结构与我方存在差异

使用适配器来消除这种差异，适配器模式的作用是：将一个接口或类转化为其他接口或类

ts
interface IUserInfo{
    getUserName:()=>string
    getHomeAddress:()=>string
}

interface IOuterUser{
    getUserBaseInfo:()=>Record<string, string>
    getUserHomeInfo:()=>Record<string, string>
}

class OuterUser implements IOuterUser{
    getUserBaseInfo(){
        return {
            username:'tom',
        }
    }
    getUserHomeInfo(){
        return {
            homeAddress:'beijing',
        }
    }
}

// extends OuterUser拿到了外包员工的数据，implements IUserInfo实现了内部接口
class OuterUserInfo extends OuterUser implements IUserInfo{
    getUserName(){
        return super.getUserBaseInfo().username
    }
    getHomeAddress(){
        return super.getUserHomeInfo().homeAddress
        
    }
}

// OuterUserInfo对象就可以在内部直接使用，因为他完整的实现了IUserInfo接口

ts
import { EventEmitter } from "events"; // 支持订阅发布的npm依赖 https://www.npmjs.com/package/events
import { interceptRequest } from "./interceptRequest";
import Util from "./Util";

export enum TrackerEvents {
  event = "event",
  jsError = "jsError",
  reqError = "reqError",
  unHandleRejection = "unHandleRejection",
  performanceInfoReady = "performanceInfoReady",
  slowHttpRequest = "slowHttpRequest"
}

export const ERROR_EVENTS: string[] = [
  TrackerEvents.jsError,
  TrackerEvents.reqError,
  TrackerEvents.unHandleRejection
];

export class Monitor extends EventEmitter {
  public static instance: Monitor;

  // 构造函数，这个例子中是使用静态方法init调用构造函数
  constructor(options: IMonitorOptions) {
    super();

    interceptRequest(this);
    
    this.observeNetworkChange()

  }

  // 单例模式，不通过构造函数创建对象，而是使用静态方法
  static init(options: Partial<IMonitorOptions>): Monitor {
    if (this.instance) {
      return this.instance;
    }

    return (this.instance = new Monitor(mergeOptions));
  }

  
 	// 总结：依赖关系可以把功能解耦，放到其他模块去执行，如果其他模块处理完还需要返回，则使用双向依赖关系，这个例子中通过订阅发布实现Monitor实例接收其他模块的回调
  // 依赖关系（引用了外部模块Util）
  public async observeNetworkChange() {
    Util.observeNetworkChange(this); // observeNetworkChange持有Monitor实例this，所以他们是双向依赖。observeNetworkChange内部通过Monitor实例的emit可以触发订阅发布
  }
  
  
  // 复写emit方法，这样所有其他模块调用 emit 方法，就会汇总到这里做处理
  override emit(event: string, data: any): boolean {
    const isErrorEvent = ERROR_EVENTS.includes(event);
    /* Check if error events hit random sample and match filter rule */
    if (isErrorEvent && !this.isEmitErrorEvent(data)) {
      return false;
    }

    /* TrackerEvents.events listen all emit events */
    super.emit(TrackerEvents.event, event, data);

    return super.emit(event, data);
  }
  


}
